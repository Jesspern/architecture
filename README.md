# ДЗ 01 – Документирование архитектуры в Structurizr (Вариант 8)

## Вариант 8 – Управление проектами

### Описание системы

Система управления проектами предоставляет пользователям возможность создавать проекты, управлять задачами, назначать исполнителей и отслеживать прогресс выполнения работ.

Основные сущности системы:
- **Проект** — контейнер для задач с участниками и сроками
- **Задача** — единица работы с исполнителем, статусом и приоритетом
- **Исполнитель** — пользователь, ответственный за задачи

Пользователи взаимодействуют с системой через веб-интерфейс. Запросы поступают через API Gateway и далее обрабатываются специализированными сервисами системы.

Основные подсистемы:
- Сервис управления пользователями (User Service)
- Сервис управления проектами (Project Service)
- Сервис управления задачами (Task Service)
- API Gateway для маршрутизации запросов

Для хранения данных используется база данных PostgreSQL.

Система интегрируется с внешними сервисами:
- Сервис отправки уведомлений (Email/SMS/Push)
- Хранилище кода (GitLab/GitHub) для связи задач с коммитами

---

### Основные API системы

Реализуются следующие операции:

**Пользователи:**
- Создание нового пользователя
- Поиск пользователя по логину
- Поиск пользователя по маске имя и фамилии

**Проекты:**
- Создание проекта
- Поиск проекта по имени
- Поиск всех проектов

**Задачи:**
- Создание задачи в проекте
- Получение всех задач в проекте
- Получение задачи по уникальному коду

---

### Роли пользователей

В системе предусмотрены следующие роли:

**Гость (Guest)**  
Незарегистрированный пользователь. Может только зарегистрироваться в системе.

**Руководитель (Tracker)**  
Пользователь с правами управления проектом. Может:
- Следить за задачами проекта
- Свободно передвигать, создавать и отменять задачи
- Управлять релизами проекта

**Исполнитель (Worker)**  
Пользователь, отвечающий за свои задачи. Может:
- Просматривать назначенные задачи
- Обновлять статус своих задач
- Создавать новые задачи

**Администратор (Administrator)**  
Пользователь с системными правами. Может:
- Следить за работой площадки
- Управлять настройками системы
- Администрировать пользователей

---

### Внешние системы

**Система уведомлений (Notification Provider)**  
Внешний сервис для отправки уведомлений пользователям:
- Email уведомления о создании задачи
- Уведомления об изменении статуса задачи
- Push/SMS уведомления

Протокол взаимодействия: HTTPS/REST

**Хранилище кода (Code Space)**  
Внешний сервис для хранения кода и связи задач с ним:
- GitLab / GitHub / Bitbucket
- Синхронизация задач с коммитами
- Привязка репозиториев к проектам

Протокол взаимодействия: Git API / HTTPS

---

### Архитектура системы

Архитектура системы описана в стиле **C4 Model** с использованием **Structurizr DSL**.

Описание архитектуры находится в файле `workspace.dsl`

---

### Диаграммы

#### C1 — System Context

Диаграмма контекста системы показывает:
- Роли пользователей (Гость, Руководитель, Исполнитель, Администратор)
- Взаимодействие пользователей с системой
- Интеграцию системы с внешними сервисами (уведомления и хранилище кода)

#### C2 — Container Diagram

Диаграмма контейнеров показывает основные компоненты системы:
- **Web UI** — пользовательский интерфейс (React/TypeScript)
- **API Gateway** — сервис маршрутизации запросов (Spring Boot)
- **User Service** — сервис управления пользователями (Spring Boot)
- **Project Service** — сервис управления проектами (Spring Boot)
- **Task Service** — сервис управления задачами (Spring Boot)
- **Database** — хранилище данных (PostgreSQL)

Также на диаграмме отображены:
- Технологии контейнеров
- Протоколы взаимодействия (HTTPS/REST, JDBC)

#### Dynamic Diagram

Динамическая диаграмма показывает архитектурно значимый сценарий:

**Создание задачи в проекте**

Сценарий включает:
1. Руководитель заполняет форму создания задачи
2. Запрос отправляется через API Gateway
3. Project Service проверяет существование проекта
4. Task Service создает новую задачу
5. Данные сохраняются в базе данных
6. Отправляется уведомление исполнителю
7. Задача привязывается к репозиторию кода
8. Ответ возвращается пользователю
9. Задача отображается в интерфейсе

---

### Используемые технологии

Основные технологии системы:

**Frontend:**
- Web UI — React + TypeScript

**Backend:**
- API Gateway — Java + Spring Boot
- User Service — Java + Spring Boot
- Project Service — Java + Spring Boot
- Task Service — Java + Spring Boot

**Database:**
- PostgreSQL — основное хранилище данных

**Взаимодействие:**
- HTTPS/REST — взаимодействие между контейнерами
- JDBC — доступ к базе данных
- Git API — интеграция с хранилищем кода

### Генерация диаграмм из Structurizr DSL

Для просмотра и экспорта диаграмм используйте расширение **C4 Architecture As A Code** в Visual Studio Code.

**Открытие диаграмм в VS Code:**
1. Установите расширение `vimpelcom.c4-varp`
2. Откройте папку с файлом `workspace.dsl`
3. Используйте команды:
   - `C4 Architecture As A Code: Focus on Architecture As A Code View`
   - Или кликните на Code Lens "Show as Structurizr Diagram" в редакторе

**Экспорт в PlantUML (Docker):**

Из корня проекта (где лежит `workspace.dsl`):

```bash
# Экспорт в PlantUML
docker run --rm -v "${PWD}":/work structurizr/cli export -workspace /work/workspace.dsl -format plantuml -output /work/puml

# Генерация PNG из PlantUML
docker run --rm -v "${PWD}":/work plantuml/plantuml -tpng /work/puml/*.puml
